# Internationalization (i18n)

## Principles

- **No hard-coded copy** in UI components for user-visible strings: use `t('...')` keys.
- **Turkish and English** ship in wave one; adding a locale means a new folder under `apps/web/src/i18n/locales/<lng>/` and registering it in `apps/web/src/i18n/index.ts`.
- **Stable keys**: prefer hierarchical keys (`pages.athletes.title`) over English sentences as keys.

## Frontend wiring

- Library: `i18next`, `react-i18next`, `i18next-browser-languagedetector`.
- Detector order: `localStorage` key `amateur.locale`, then browser language.
- Language switch: `LanguageSwitch` component in the app header and landing page.

## Backend

- API error messages for end users will be localized in a later wave (e.g. error codes + client-side maps, or `Accept-Language`).
- Entity display names in the DB remain neutral; UI labels stay in locale files.

## Adding a language

1. Create `apps/web/src/i18n/locales/<lng>/common.json` (copy from `en` as template).
2. Add `lng` to `supportedLngs` and `resources` in `apps/web/src/i18n/index.ts`.
3. Add a button in `LanguageSwitch` (or derive options from `supportedLngs` in a later refactor).
